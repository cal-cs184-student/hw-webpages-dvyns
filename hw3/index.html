Part 1: Ray Generation and Scene Intersection

Ray Generation and Intersection

For each pixel, ideally we want the color to correspond to the integral of the radiance over that pixel. To estimate this, we generate a number of sample rays that intersect that pixel's area and find the nearest object (or objects, if we encode transparency) each of those rays intersect. This approach is motivated by Monte Carlo integration. At each ray intersection, we estimate how much light is reflected into the camera from that point. Summing this all up and normalizing gives us a decent estimate of the pixel color.

For this part we encode the basic logic for generating these rays and program some basic tests for intersection with primitives in the scene. Because it is expensive the check for intersection with all primitives, we will later use a bounding boxes to limit how many of these tests we are doing.

Task 1: Generating Camera Rays

In this part we note that the input \((x, y)\in [0, 1]\times[0,1]\) and we want the \(x\) and \(y\) components of the output \((x_c, y_c)\) in camera space to be in the range \[(x_c, y_c) \in \left[-\tan\left(\frac{hFov}{2}\right), \tan\left(\frac{hFov}{2}\right)\right]\times \left[-\tan\left(\frac{vFov}{2}\right), \tan\left(\frac{vFov}{2}\right)\right]\]. So construct a mappig that will take 0 to \(-\tan\left(\frac{hFov}{2}\right)\) and 1 to \(\tan\left(\frac{hFov}{2}\right)\) for the \(x\) component, and similarly for the \(y\) component linearly interpolating for points in between. We write this as: 

cam_space_x = 2.0 * (x - 0.5) * tan(radians(this->hFov) / 2.0)
cam_space_y = 2.0 * (y - 0.5) * tan(radians(this->vFov) / 2.0)

We place these points in the plane at $z=-1$ as a Vector3D object. Then we multiply this vector by the c2w transformation matrix to get the final ray direction, normalize it to unit length, and assign this direction, position, min_t, and max_t to the returned ray.

Task 2: Generating Pixel Samples

We loop through nsaa sampling from GridUniformSampler2D() at each point. Each sample, which is between 0 and 1 in both directions, is added to the position $(x, y)$ to get the location within the corresponding pixel. We then normalize by the buffer width in the $x$ direction and buffer height in the $y$ direction so that $x+x_{sample}$ and $y+y_{sample}$ are normalized.

This normalized sample is then passed into the generate ray function from Task 1. The resulting ray from camera->generate_ray() is passed into est_global_radiance(). We accumulate these estimated radiances and normalize by nsaa. We then update the pixel color with this accumulation.

Task 3: Ray Triangle Intersection

For this task, we implemented the Möller-Trumbore algorithm as described in the ray-tracing slides. Instead of directly doing the scalar multiplication \[ \frac{1}{S_1\cdot E_1}\begin{bmatrix} S_2\cdot E_2 \\ S_1\cdot S \\ S_2\cdot D\end{bmatrix} \] with vectors, we calculated each point separately, since each element of the resulting vector has a separate condition necessary for intersection. Doing this saves computation when earlier checks fail.

$t = S_2\cdot E_2 / (S_1\cdot E_1)$: The time of intersection $t$ must be in between r.min_t and r.max_t. If it isn't we return false.

$b_1 = S_1\cdot S / (S_1\cdot E_1)$: The barycentric coordinate $b_1$ must be in the range $[0, 1]$. Otherwise the point lies outside the trieangle and we return false.

$b_2 = S_2\cdot D / (S_1\cdot E_1)$: The sum of barycentric coordinates $b_1 + b_2$ must be in the range $[0, 1]$. Otherwise the point lies outside the triangle, and we return false.

If all of these conditions are met, than the given ray intersects the interior of the current triangle at time $t$. In Triangle::has_intersection() we are done. For Triangle::intersect() we still need to assign values to the intersection object: 

isect->t = t;
isect->n = ((1 - b_1 - b_2) * n1 + b_1 * n2 + b_2 * n3).unit();
isect->primative = this;
isect->bsdf = this->get_bsdf();
Then we need to update r.max_t = t. (We forgot to do this and the banana looked all messed up.)

Part 2: Bounding Volume Hierarchy

For constructing our bounding volume hierarchy, we used an exhaustive surface area heuristic. First we check to make sure that we have more than max_leaf_size primitives in our current node. If not, this is a leaf and we return the current node. Otherwise we move on to constructing partitions.

There, on each axis, we first sort all primitves into a vector ordered by their centroids axis position. Then we construct splits for each point along each axis into two vectors. For the first_half vector, we have: 
std::vector first_half(n);
    first_half[0] = sorted[0]->get_bbox();
    for (size_t i = 1; i < n; i++) {
      first_half[i] = first_half[i - 1];
      first_half[i].expand(sorted[i]->get_bbox());
    }
This chooses the furthest "left" primitive, gets its bounding box, and successively expands it for each additional point. The second_half vector does the same in reverse, starting from the furthest "right" primitive. These definitions mean that the set of primitives contained in the union of first_half[split-1] and second_half[split] is mutually disjoint and contains all primitives, as desired from a split.

We keep track of the best split and best axis based on the cost function 

double cost = C_trav + (first_half[split - 1].surface_area() * split + 
                        second_half[split].surface_area() * (n - split)) / 
			bbox.surface_area();
where C_trav = 0.125, the estimated traversal cost from lecture. To clarify, split is an iterator over the sorted vector of primitives whose position determines the split point.

All of this is done on a copy of the vector of primitives with sorting handled by std::sort(). We check that best_cost is less than number of primitives $n$, the estimated relative cost of iterating through all primitives in the node. If it is greater than $n$ we estimate a leaf to be more efficient than any split and return the node.

Otherwise, we sort the real vector of primitives (calling std::sort() directly on start and end) along the best_axis in order of centroid position on that axis. Then we recurse on both splits (using the convention that end is exclusive): 

auto mid = start + best_split;
node->l = construct_bvh(start, mid, max_leaf_size);
node->r = construct_bvh(mid, end, max_leaf_size);

The function returns the current node.

Our intersection algorithms work pretty much as would be expected. For the bounding box, we check each of the $x,y,z$ planes and use the minimum time intersection if there is one. We iterate through the BVH in the same way you would a recursive tree traversal. The has_intersection function stops uses a logic shortcut to avoid doing unnecessary work.

The performance gains of our BVH scale with the size/density of the meshes. Max Planck, which is the most complex scene, shows the most significant improvement. This is expected, since tree searching is roughly $O(\log n)$, where $n$ is the number of primitives whereas the naive approach is approximately $O(n)$. There are some edge cases to this, but our performance improvement should approach that asymptotic behavior as the mesh gets large enough.

The other significant point to note, however, is that the BVH construction time also scales with mesh size. The BVH construction times are not that noticeable with these meshes, but when rendering something like wall-e.dae...

and we see that the BVH construction time dwarfs our rendering time by an order of magnitude!

Assume a reasonably well balanced tree, which should be a reasonable assumption in this case. Then, for the construction time, we have the recurrence relation \[ T(n) = 2T(n/2) + O(n\log n) \implies T(n) = O(n\log^2 n). \] (This is because runtime is dominated by the $O(n\log n)$ sorting at each level).

If we instead used a bucketed approach, like in PBRT, we would only need to partition the primitives: this is an $O(n)$ operation. Hence, from the recurrence, we would get an asymptotic bound $T(n)=O(n\log n)$. This is significantly better, and suggests we would probably be better off using a bucketed approach.

Part 3: Direct Illumination

Uniform Hemisphere Sampling (estimate_direct_lighting_hemisphere)

To estimate direct lighting using uniform hemisphere sampling, we sample num_samples = scene->lights.size() * ns_area_light random directions uniformly over the hemisphere at the hit point. For each sample:

  • Sample a random direction wi in object space using hemisphereSampler->get_sample()
  • Transform wi to world space using the o2w matrix
  • Cast a shadow ray from hit_p in that direction with min_t = EPS_F to avoid self-intersection
  • If the shadow ray intersects an emissive surface, retrieve its emission via get_emission()
  • Accumulate the contribution: f(w_out, wi) * emission * cos(theta) / pdf

Since we sample uniformly, pdf = 1/(2π), so we multiply by 2π.
cos(theta) = wi.z since z is "up" in object space.

Average over all samples by dividing by num_samples.

Importance Sampling (estimate_direct_lighting_importance)

Instead of sampling random directions, we sample directly toward each light source:

  • Loop over every light in the scene
  • For delta lights (point lights), take 1 sample; for area lights, take ns_area_light samples
  • For each sample, call light->sample_L(hit_p, &wi_world, &dist, &pdf) to get:
    • A direction toward the light wi_world
    • The distance to the light dist
    • The probability density pdf
  • Transform wi_world to object space and skip if wi.z < 0 (light behind surface)
  • Cast a shadow ray with max_t = dist - EPS_F if nothing blocks it, the light is visible
  • Accumulate: f(w_out, wi_obj) * radiance * cos(theta) / pdf

Average each light's contribution over its samples, then sum across all lights.

Hemisphere sampling wastes most samples on directions that don't hit any light, while importance sampling guarantees every sample contributes useful lighting information by sampling directly toward lights.

Noise Level Comparison: Hemisphere vs. Importance Sampling

Uniform hemisphere sampling and importance sampling produce noticeably different results for the same scene and sample count. With hemisphere sampling, the render is significantly noisier, the walls, floor, and bunny surface all exhibit heavy grainy speckles because most sampled directions miss the light source entirely, contributing nothing to the estimate. The soft shadows are poorly defined and the overall image appears darker and less converged. Importance sampling, by contrast, produces a much cleaner render with the same settings soft shadows under the bunny are smooth and well-defined, color bleeding from the red and blue walls is more accurate, and the overall lighting looks physically correct. This difference arises because importance sampling guarantees every shadow ray is cast toward a light source, making each sample maximally informative, whereas hemisphere sampling wastes the vast majority of samples on directions that carry no direct lighting contribution. In short, importance sampling achieves the same visual quality as hemisphere sampling with far fewer samples, making it the clearly superior method for direct lighting estimation in scenes with small area lights.

Part 4: Global Illumination

We extended our Part 3 estimator into a recursive global-illumination integrator. All Part 4 images below are full-scene 640x480 renders rather than cropped window renders.

Implementation Walkthrough

The starting point is the standard decomposition of outgoing radiance into emitted, direct, and indirect terms:

\[ L_o(x, \omega_o) = L_e(x, \omega_o) + L_{\text{direct}}(x, \omega_o) + L_{\text{indirect}}(x, \omega_o). \]

Our top-level integrator follows that split directly. We first intersect the camera ray against the BVH, return environment radiance when nothing is hit, add emitted radiance at the primary hit, and then add the recursive bounce estimator if the ray still has depth remaining. 

Vector3D PathTracer::est_radiance_global_illumination(const Ray &r) {
  Intersection isect;
  Vector3D L_out;

  if (!bvh->intersect(r, &isect))
    return envLight ? envLight->sample_dir(r) : L_out;

  L_out = zero_bounce_radiance(r, isect);

  if (max_ray_depth > 0) {
    L_out += at_least_one_bounce_radiance(r, isect);
  }
  return L_out;
}

Here, zero_bounce_radiance represents the emitted term L_e, while at_least_one_bounce_radiance is responsible for the transport beyond pure emission. This means Part 4 builds on top of Part 3 instead of replacing it.

Inside the recursive helper, the indirect term is estimated with Monte Carlo sampling over the BSDF. In integral form, the bounced contribution is

\[ L_{\text{indirect}}(x, \omega_o) = \int_{H^2} f_r(x, \omega_i, \omega_o)\,L_o(x', -\omega_i)\,|\cos \theta_i|\,d\omega_i. \]

After sampling one direction from the BSDF, our one-sample estimator becomes

\[ \widehat{L}_{\text{indirect}} = \frac{f_r(x, \omega_i, \omega_o)\,L_o(x', -\omega_i)\,|\cos \theta_i|}{p(\omega_i)\,p_{rr}}, \]

where p(omega_i) is the BSDF sampling density and p_rr = 0.7 is the Russian roulette survival probability. That estimator appears almost verbatim in our recursive implementation. 

Vector3D PathTracer::at_least_one_bounce_radiance(const Ray &r,
                                                  const Intersection &isect) {
  Matrix3x3 o2w;
  make_coord_space(o2w, isect.n);
  Matrix3x3 w2o = o2w.T();

  Vector3D hit_p = r.o + r.d * isect.t;
  Vector3D w_out = w2o * (-r.d);
  Vector3D L_out(0, 0, 0);

  if (isAccumBounces || r.depth == 1) {
    L_out += one_bounce_radiance(r, isect);
  }
  if (r.depth <= 1) return L_out;

  double ppdf = 0.7;
  if (!coin_flip(ppdf)) return L_out;

  Vector3D w_obj;
  double pdf;
  Vector3D f = isect.bsdf->sample_f(w_out, &w_obj, &pdf);
  if (pdf == 0) return L_out;

  Vector3D w_world = o2w * w_obj;
  Ray next_ray(hit_p, w_world);
  next_ray.min_t = EPS_F;
  next_ray.depth = r.depth - 1;

  Intersection nxt_isect;
  if (bvh->intersect(next_ray, &nxt_isect)) {
    Vector3D incoming = at_least_one_bounce_radiance(next_ray, nxt_isect);
    if (isect.bsdf->is_delta()) {
      incoming += zero_bounce_radiance(next_ray, nxt_isect);
    }
    L_out += f * incoming * abs_cos_theta(w_obj) / (pdf * ppdf);
  }
  return L_out;
}

Several details matter here. We sample the BSDF in the local shading frame, so the outgoing direction is converted with \( \omega_o^{\text{local}} = T^{-1}(-r.d) \), the sampled incoming direction is mapped back with \( \omega_i^{\text{world}} = T\,\omega_i^{\text{local}} \), and the cosine term is accumulated as abs_cos_theta(w_obj). The next_ray.min_t = EPS_F line prevents self-intersections, and the explicit bvh->intersect(next_ray, &nxt_isect) test ensures that recursion only continues when a real scene hit exists.

Russian roulette keeps the recursion unbiased. If a random variable X represents the surviving bounce contribution, then

\[ \mathbb{E}\left[\frac{\mathbf{1}_{\text{survive}}}{p_{rr}} X\right] = \mathbb{E}[X]. \]

In practice, this means long low-energy paths can terminate early, while the surviving paths are reweighted by 1 / ppdf so the estimator still converges to the correct value.

The delta-material case needs one more correction. For mirror and glass paths, the next hit may be an emissive light mesh. At that point the radiance arriving from the next vertex is

\[ L_o(x', -\omega_i) = L_e(x', -\omega_i) + L_{\text{bounce}}(x', -\omega_i). \]

Our recursive helper naturally returns the bounced term, but not the emitted term at that recursive hit, so the delta branch explicitly adds zero_bounce_radiance(next_ray, nxt_isect). That is what makes light sources show up correctly in specular reflections and refractions.

Part 5: Adaptive Sampling

Standard path tracing uses a fixed number of samples per pixel regardless of visual complexity. Adaptive sampling improves efficiency by detecting when a pixel has statistically converged and stopping early, spending more samples only where they are genuinely needed, such as shadow boundaries and complex indirect lighting, and fewer on flat diffuse surfaces.

Convergence Test

For each pixel, we track two running sums over the per-sample luminance values:

s1 = Σ xₖ      s2 = Σ xₖ²

From these we compute the mean and variance:

μ = s1 / n      σ² = (1 / (n - 1)) (s2 - (s1² / n))

And the 95% confidence interval half-width:

I = 1.96 · (σ / √n)

The pixel is declared converged when:

I ≤ maxTolerance · μ

This is a relative threshold, a bright pixel can tolerate more absolute variance than a near-black one, naturally matching human visual perception.

Implementation

For each pixel, we loop up to num_samples times, accumulating radiance and tracking running statistics. Every samplesPerBatch (16) samples we check the convergence condition and break early if met: 

	initialize s1, s2, m = 0
	
	for each sample up to num_samples:
		generate jittered camera ray through pixel
		trace ray → get radiance
	
		accumulate radiance into L_out
		compute luminance of sample → illum
		update s1 += illum, s2 += illum²
		m++
	
		if m is a multiple of samplesPerBatch:
			compute mean, variance, and I from s1, s2, m
			if I ≤ maxTolerance × mean → break (converged)
	
	write L_out / m to pixel
	write m to sampleCountBuffer

We use luminance as the convergence scalar rather than per-channel RGB, reducing the measurement to a single perceptually weighted value. The actual sample count m is stored in sampleCountBuffer, which drives the false-color sample rate image, blue pixels converged early with few samples, while red pixels required close to the full 2048 samples.
