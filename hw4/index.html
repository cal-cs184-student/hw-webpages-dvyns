Overview

In this homework, we built a cloth simulation and rendering pipeline. We represented the cloth as a mass-spring system. Verlet integration was used to update the positions of the masses based on structural, shearing, and bending forces. We then handled collisions with planes, spheres, and itself through spatial hashing to check that masses are in valid positions. When they are not, we apply a correction vector to push it back to a valid spot. We the implemented a set of shaders: diffuse lighting, Blinn-Phong shading, texture mapping, bump mapping, displacement mapping, and environment-mapped reflections.

Part 2: Simulation via numerical integration

Default Parameter Simulation

Effects of Changing ks (Spring Constant)

With a very low ks (50 N/m), the springs offer little resistance to deformation. The cloth shows far more ripples and never fully smooths out even at rest. Although it cannot be seen in the image oscillations continue for much longer than in the default case.

With a very high ks (50000 N/m), the springs strongly resist deformation and the cloth hangs much more rigidly. As seen in the image, the top edge forms only a singlecurve with minimal wrinkling. The overall surface is mostly flatter compared to the default parameters.

Effects of Changing Density

With low density (1 g/cm²), the cloth does not have enough weight to fully extend most of the springs. It hangs with a shallower smooth curve at the top and a relatively flat body.

With high density (150 g/cm²), the cloth to sags much deeper. The there are multiple curves that form at the top and sides pull inward. The resulting image is more visually similar to the low-ks case since both result in greater deformation relative to the spring forces.

Effects of Changing Damping

With damping at 0%, energy is never removed from the system and it continues to oscillate chaotically never reaching a resting state.

With damping at ~1% (damping1), the cloth loses energy quickly and settles into a smooth, stable resting position with no oscillation, falling in a slow controlled motion compared to the undamped case. It looks very similar to the default parameter case.

Part 3: Handling collisions with other objects

Sphere Collisions

Each point mass checks whether its updated position lies inside the sphere. If it does, the code computes a direction from the sphere center toward the point mass, projects the point to the sphere surface, and moves the mass from last_position toward that surface point using a friction-scaled correction.

The tangent point is computed by extending the ray from the origin through the point mass position to land exactly on the sphere surface. A correction vector is computed from the point mass's last position to this tangent point and applied scaled by (1 - friction).

The implementation also guards against the degenerate case where the point mass lands exactly at the sphere center by falling back to the previous-step direction instead of normalizing a zero vector.

The core sphere collision implementation is shown below: 

void Sphere::collide(PointMass &pm) {
  Vector3D diff = pm.position - origin;
  double distance = diff.norm();
  if (distance >= radius) {
    return;
  }

  Vector3D direction;
  if (distance > 0) {
    direction = diff / distance;
  } else {
    Vector3D last_diff = pm.last_position - origin;
    direction = last_diff.norm() > 0 ? last_diff.unit() : Vector3D(0, 1, 0);
  }

  Vector3D tangent_point = origin + direction * radius;
  Vector3D correction = tangent_point - pm.last_position;
  pm.position = pm.last_position + correction * (1.0 - friction);
}

Plane Collisions

For plane collisions, each point mass is tested by computing the signed distances of its current and last positions from the plane using the dot product with the plane normal. If the signs differ, the point mass has crossed the plane between timesteps. It avoids falsely re-colliding a point mass that starts exactly on the plane and then moves away from it.

When a correction is needed, the code projects the point mass to the plane, nudges it by a small SURFACE_OFFSET to avoid clipping, and applies the correction from last_position with friction. The offset direction is chosen so exact-contact edge cases are pushed to the correct side of the plane.

The plane collision code: 

#define SURFACE_OFFSET 0.0001

void Plane::collide(PointMass &pm) {
  double curr_dist = dot(pm.position - point, normal);
  double last_dist = dot(pm.last_position - point, normal);

  if ((curr_dist > 0 && last_dist > 0) ||
      (last_dist == 0 && curr_dist >= 0)) {
    return;
  }

  Vector3D tangent_point = pm.position - normal * curr_dist;
  Vector3D offset_direction;
  if (last_dist > 0) {
    offset_direction = normal;
  } else if (last_dist < 0) {
    offset_direction = -normal;
  } else {
    offset_direction = curr_dist < 0 ? normal : -normal;
  }

  Vector3D correction =
      tangent_point + SURFACE_OFFSET * offset_direction - pm.last_position;
  pm.position = pm.last_position + correction * (1.0 - friction);
}

Part 4: Handling self-collisions

We implemented self-collision with spatial hashing in Cloth::hash_position, Cloth::build_spatial_map, and Cloth::self_collide. At each simulation step, the cloth partitions space into boxes of size 3 * width / num_width_points by 3 * height / num_height_points by max(w, h). Each point mass is hashed into one of these boxes, and the spatial map stores a list of point masses for each occupied box.

During self-collision resolution, each point mass only checks the other point masses in its own hash bucket instead of comparing against the entire cloth. If another mass is closer than 2 * thickness, we compute the displacement needed to push the current mass away until the pair is exactly 2 * thickness apart. The final displacement is the average of all such correction vectors divided by the number of simulation steps, which keeps the correction from being too abrupt in a single step. Pinned masses are not moved by this pass.

The spatial map construction and hash function are shown below. The cloth bins each point mass into a 3D box whose dimensions are based on the cloth resolution, then uses a weighted combination of the snapped box coordinates as the hash key: 

void Cloth::build_spatial_map() {
  for (const auto &entry : map) {
    delete(entry.second);
  }
  map.clear();

  for (auto &pm : point_masses) {
    float key = hash_position(pm.position);
    if (map.find(key) == map.end()) {
      map[key] = new vector<PointMass *>();
    }
    map[key]->push_back(&pm);
  }
}

float Cloth::hash_position(Vector3D pos) {
  double box_width = 3.0 * width / num_width_points;
  double box_height = 3.0 * height / num_height_points;
  double box_depth = box_width > box_height ? box_width : box_height;

  double x = pos.x - fmod(pos.x, box_width);
  double y = pos.y - fmod(pos.y, box_height);
  double z = pos.z - fmod(pos.z, box_depth);

  return x * 73856093.0 + y * 19349663.0 + z * 83492791.0;
}

The actual self-collision response then looks only within the current point mass's bucket, accumulates separation corrections against nearby masses, and applies the average displacement scaled by the number of simulation steps: 

void Cloth::self_collide(PointMass &pm, double simulation_steps) {
  if (pm.pinned) return;

  float key = hash_position(pm.position);
  if (map.find(key) == map.end()) return;

  Vector3D correction(0, 0, 0);
  int num_corrections = 0;
  double min_distance = 2.0 * thickness;

  for (PointMass *candidate : *map[key]) {
    if (candidate == &pm) continue;

    Vector3D direction = pm.position - candidate->position;
    double distance = direction.norm();
    if (distance >= min_distance) continue;

    if (distance == 0) {
      direction = pm.position - pm.last_position;
      if (direction.norm() == 0) {
        direction = Vector3D(1, 0, 0);
      }
      distance = 0;
    }

    correction += direction.unit() * (min_distance - distance);
    num_corrections++;
  }

  if (num_corrections > 0) {
    pm.position += correction / num_corrections / simulation_steps;
  }
}

Changing density changes how strongly the spring forces accelerate each point mass. Lower density makes the cloth respond more strongly to the same spring forces, so folds tend to open up and rebound more. Higher density makes the cloth heavier relative to its springs, so it collapses into deeper, more compressed folds. In practice, lowering the density produces a similar appearance to increasing the spring constant and vice versa, which is what we would expect.

Part 5: Shaders

Shaders are a collection of programs that run parts of the graphics pipeline in pararallel on a GPU. This is done to decrease the required runtime to render scenes, especially real-time scenes. Vertex shaders transform attributes of the vertices of an object based on some underlying map. Following rasterization, fragment shaders use the vertex shader values to interpolated corresponding values for each pixel in a fragment.

Blinn-Phong shading

Blinn-Phong shading is a lighting model that tries to approximate the light scattering behavior of various surfaces. It does this by calculating three different terms: the ambient term, the diffuse term, and the specular term. The ambient term is based on some small constant light emanating from a surface. The diffuse term is what it sounds like, essentially represents the diffuse reflection from a diffuse surface. The specular term adds highlights of mirror like reflections. The relative ratio/strength of these three terms helps to immitate real reflective behavior.

The four renders below show the same Blinn-Phong shader with different output terms enabled: blinnphong1.png is ambient only, blinnphong2.png is diffuse only, blinnphong3.png is specular only, and blinnphong4.png is the full ambient + diffuse + specular result. These were generated by temporarily outputting the corresponding term from Phong.frag and then restoring the full sum for the final render.

Texture mapping

The vertex shader passes the interpolated UV coordinate into the fragment stage, and Texture.frag samples u_texture_1 with texture(u_texture_1, v_uv), so the visible color comes from an image lookup instead of from a constant color. To make texture selection easier, we updated clothSimulator.h and clothSimulator.cpp to scan the /textures directory at startup, populate a texture dropdown in the GUI, and reload the selected image into the texture unit used by Texture.frag. That means we can add another supported image file to /textures and choose it from the GUI without editing the shader or hardcoding another texture path.

Bump mapping and displacement mapping

Bump mapping (Normal = 10, Height = 2)

Displacement mapping (Normal = 10, Height = 2)

Both shaders, bump mapping and displacement mapping, use u_texture_2 as a height map. We convert the sampled RGB value to a scalar height with a Rec.709-style luminance dot product. For bump mapping, we estimate the height gradient in texture space with: 

dU = (h(u + 1 / w, v) - h(u, v)) * k_h * k_n
dV = (h(u, v + 1 / h) - h(u, v)) * k_h * k_n

We use those finite differences to build the perturbed normal (-dU, -dV, 1) in tangent space and transform it with the TBN matrix. Bump mapping changes only the shading normals, while displacement mapping also moves each vertex along its original normal in Displacement.vert. That is why, as shown in the screenshots above, bump mapping preserves the original silhouette of the objects but displacement mapping doesn't. If we were to use something closer to the default settings shown in the spec on the displacement map we would have a far less perturbed geometry. We opted not to do that to make the vertex displacement more obvious.

To make the required comparisons easier on us, we added a Height Map dropdown to the Appearance panel. It, like our texture dropdown menu implementation, scans the /textures directory and reloads the selected file into u_texture_2. That way we can switch our bump/displacement map textures without having to finagle around the codebase.

With Normal = 20 and Height = 1, bump mapping changes far less compared to the displacement mappingas the sphere gets finer because it is perturbing only the shading normals; the coarse and fine renders mainly differ near the silhouette. Even then, it is barely perceptible.

Displacement mapping reacts more dramatically to the same coarseness change because the geometry itself is being altered. At -o 16 -a 16 the displaced relief looks more like a low-poly mesh while at -o 128 -a 128 the same texture produces a much more detailed surface reflecting more of the detail on the underlying image.

Environment Mapped Reflections

The spec doesn't ask for an explanation of this part but we will give one just in case. The mirror shader computes the direction from the fragment to the camera, reflects that direction about the surface normal, and uses the reflected direction to sample the cubemap. That cubemap lookup provides the reflected environment color for the fragment. This doesn't work all that well for non curved surfaces, but when you have a curved surface it is reasonably convincing.
